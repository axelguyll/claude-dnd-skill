# Findings — D3: Architecture coherence (do fragments hold across a playthrough?)

*Fable review, Session E — 2026-07-16. Bootstrap: ARCHITECTURE.md, CONTEXT.md, compass,
D1/D2 handoffs. Method: full lifecycle trace (load → scenes → combat → save → compaction →
reload) against the actual prose procedures and scripts, plus the four checks Sessions C/D
handed to E. Propose-only per ground rules — all fixes batch in Session F.*

Verdict up front: the compaction-survival **design** is genuinely good (anchors, ladder,
micro-save, deterministic backstops), but three of its load-bearing wires are unplugged:
the session counter that every time-stamped mechanism keys off is never incremented, the
one narrative anchor that stays fresh mid-session (the session tail) has no reader, and
mid-combat state — the exact place long sessions hit compaction — has no durable home.
The four handoff checks all resolve: D2's fixes are safe to apply **with the riders below**.

---

## HIGH severity

### E1 — Nothing ever increments `Session count` (or `Last session`)

**Claim.** The state.md header field `**Session count:**` is seeded `0` by `new`
(SKILL-commands.md:61) and `prep` (SKILL-commands.md:329) and then **no procedure ever
increments it**. `/dm:dnd save` (SKILL-commands.md:366-438) updates location, quests,
flags, moves, deeds, tail, archive, graph — the header is never mentioned. `/dm:dnd end`
(SKILL-commands.md:442-457) adds recap/calibration/world-state/arc steps — still no
header touch. Same for the `**Last session:**` date.

**Why HIGH — the blast radius is every time-stamped mechanism:**
- Graph edges: every `add-edge` must carry `--since <current-session-N>` read from the
  header (SKILL-commands.md:120, :436, :756). Count stuck at 0 → all edges "since
  session 0" → the time-stamped historical replay the graph exists for
  (SKILL-commands.md:739) is flat.
- Log archival trigger: "run on every save after session count > 3"
  (SKILL-commands.md:391) — never fires; session-log.md grows unbounded, defeating the
  lazy-load design.
- Legacy detection: "`Session count > 1`" (SKILL-commands.md:126) — a 20-session campaign
  still reads as fresh, skipping the graph-init backup/compression path built for it.
- `name_registry.py` `first_session` (name_registry.py:94-99), `npc_rename.py` audit
  session stamps (npc_rename.py:293-297), Continuity Archive `### Session N` labels,
  `scene-context --at-session` — all keyed to the same dead counter.

In practice the model may *infer* a bump when writing a `## Session N` log entry — which
is worse than deterministic failure: `end` runs `save` plus extras, and a mid-session
`save` followed by `end` invites double-increment; two saves in one sitting invite two
log entries for one session. The session boundary is simply undefined.

**Proposed fix (prose, save step 1 region, SKILL-commands.md:366-368):** define the
boundary once: *"If this save is creating a new `## Session N` entry in session-log.md
(i.e. the first save since load), set N = header `Session count` + 1, write the entry
under that N, and update the header: `Session count: N`, `Last session: <today>`. Later
saves in the same sitting update the existing `## Session N` entry and do not bump the
header."* One paragraph; every downstream consumer starts working.

### E2 — The session tail is the only anchor micro-save keeps fresh, and nothing reads it

**Claim.** The compaction re-read ladder (SKILL.md:213-221) sends recap claims to Live
State Flags → `## Current Situation`/`## Recent Events` → npcs-full → Continuity Archive
→ session-log. But `## Recent Events` and `## Current Situation` are **save-time**
artifacts — the micro-save flush list (SKILL.md:223) updates Live State Flags, graph, and
*the session tail*, not Recent Events. So after a mid-session compaction, the freshest
narrative record in existence is `session-tail.md`/`session_tail.json` — refreshed every
scene boundary — and **no step anywhere reads it**: not the ladder, not `/dm:dnd load`
(SKILL-commands.md:98-110), not the recap. It is written at save (SKILL-commands.md:386-389)
and verified at end (SKILL-commands.md:457) purely for its own sake.

History confirms this is an amputation, not a design: the tail's reader was the display
companion's load-time replay (CHANGELOG.md:336 — "`/dnd load` had no last-scene replay to
play back"); the 2026-07-15 teardown kept the writer ("survives as a continuity file",
CHANGELOG.md:20) and removed the only consumer. This **answers the compass open question**
("who is session_tail.json for — future hook or dead weight?"): neither — it's the right
artifact with a severed wire. Wire it, don't cut it.

**Proposed fix (prose, two insertions):**
1. Re-read ladder, after the Live State Flags stop (SKILL.md:214): *"**For what just
   happened this session (post-compaction):** read `session-tail.md` — it is refreshed at
   every micro-save and is the freshest narrative record; `## Recent Events` only catches
   up at full save."*
2. Load step 5 (SKILL-commands.md:98): add `session-tail.md` to the read list (it's 5-8
   bullets — cheap) so the opening recap can name the final stretch of last session, which
   `## Recent Events` describes only in summary.

### E3 — Mid-combat compaction loses the fight: `## Active Combat` is written once, at combat start

**Claim.** `combat start` step 3 saves the initiative STATE_JSON to `state.md → ## Active
Combat` (SKILL-commands.md:661). The per-turn sequence (SKILL.md:288-303) then updates
tracker.json (conditions/concentration/effects only), re-renders `tracker.html` with a
fresh STATE_JSON passed *from context* (SKILL.md:296-299), and persists sheets only "at
scene boundaries / autosave cadence" (SKILL.md:301-302). Nothing writes the evolving
STATE_JSON back to `## Active Combat`. So mid-fight, current NPC HP, round number, and
turn order live **only in context** (and in tracker.html, which the ladder never
mentions). Long combats are precisely where compaction lands; on compaction at round 6,
the recovery source says round 1 with everyone at full HP. The re-read ladder
(SKILL.md:213-221) has no combat entry at all, so the model won't even be pointed at the
stale copy — it will trust the compacted impression the ladder exists to distrust.

**Proposed fix (prose):**
1. Per-turn step d2 (SKILL.md:296-299), one added sentence: *"After the render, write the
   same STATE_JSON back to `state.md → ## Active Combat` (replace the block) — the render
   and the durable copy must never diverge."* The JSON is already in hand; this is one
   Edit per round.
2. Ladder addition (SKILL.md:213-221): *"**Mid-combat:** re-read `state.md → ## Active
   Combat` for order/HP/round and run `tracker.py -c <camp> list` for
   conditions/concentration — never reconstruct a fight from compacted memory."*

## MEDIUM severity

### E4 — E-check 1 (D2-1 fix 4): the authored `arc revise` spine write-back — verdict: **required, but only with an ordering contract**

**Trace of both dual-writes as handed off:**
- *Existing:* `beat complete` writes spine.json (status flips, step 1,
  SKILL-commands.md:350-351) then mirrors to state.md (step 2, :352-356). Direction
  spine→state, no validator (compass [A]). Interruption between steps 1 and 2 leaves
  spine and mirror disagreeing on `current_beat`; a replayed `beat complete` can re-run
  step 3's leveling loop (the milestone marker is an absolute level, which mostly
  protects against double-leveling, but "once per level" from a stale sheet state is
  not fully idempotent).
- *Proposed (D2-1 fix 4):* `arc revise` writes state.md then back-writes spine.json —
  the **opposite** direction. Two opposed half-syncs mean neither store is ever simply
  authoritative: after a revise the mirror is fresher; after a complete the spine is.
  A future validator can't resolve a mismatch by rule.

**The interaction that settles it:** D2-2 fix 1 has `beat complete` read the new current
beat's `situation`/`threats`/`secret` **from spine.json** into `steering_notes`. If
revise does *not* write back, every post-revision beat re-arms the table with
pre-revision content — silently resurrecting the exact D2-2 failure. So once D2-2 lands,
the write-back is not optional. F should apply it **shaped like this** (revise flow,
SKILL-commands.md:723-733, authored branch):

1. Write the revised `what_changes`/`world_pressure` to the matching beat in
   **spine.json first**.
2. Re-run the schema gate: `prep/schema.py --bible .../spine.json` (revise only touches
   two prose fields, so this always passes unless the edit broke structure — which is
   exactly when you want to know).
3. Regenerate the state.md mirror entry **from the just-written spine**, not from
   context — the sync is then always spine→state, one direction, and the spine stays the
   sole prose authority.
4. Append `revision_log` in state.md (spine.json gets no new fields; schema.py ignores
   unknown keys but there's no reason to add any).

**Plus the cheap validator both dual-writes have been missing:** `beat complete` gains a
step 0 — *"compare the mirror's `beats[].status` + `current_beat` against spine.json
before advancing; on any mismatch, stop, show both, and reconcile with the host — do not
silently pick a winner."* Deterministic drift detection at the only moment both files are
open anyway.

### E5 — E-check 2 (D2-2): `steering_notes` consumers — verdict: **payload is safe; one spoiler rider**

Grep across prose, scripts, templates, tests: **no script parses `steering_notes`.**
Consumers are all prose — load reads it inside `## Campaign Arc`
(SKILL-commands.md:105), the detour rule (SKILL.md:235) and dynamic rule 4 (SKILL.md:253)
consume it as guidance, end step iv refreshes it (SKILL-commands.md:456), `arc
advance`/`revise` rewrite it (:701, :732). The only automated touch is
tests/test_prep_lifecycle.py:27,68 asserting the *key exists* — payload-agnostic. The
template field is a YAML folded block (`steering_notes: >`, templates/state.md:169), so a
3-item multi-line payload (situation / threats / secret) parses fine.

**The one real interaction:** `/dm:dnd arc status` prints `steering_notes` verbatim to
whoever asks (SKILL-commands.md:700) — and in solo play the host *is* the player. Today
that already exposes `what_changes` (arc status is established DM-eyes); adding the
beat's `secret` upgrades the leak from "structure" to "the one field that exists purely
to be discovered in play". **Rider for F:** the D2-2 payload writes the secret line as
the last line, and `arc status` (authored) gains: *"omit the `secret` line when
printing steering_notes."* One sentence each.

### E6 — E-check 3 (D2-7): re-reading SKILL.md post-compaction — verdict: **feasible, but only if the skill-dir path is anchored somewhere durable**

The mechanism itself is ordinary: SKILL.md is a file on disk, the Read tool re-reads it
exactly like a campaign file. The gap is the **path**. Campaign files hang off a
derivable root (`~/.claude/dnd/` or `$DND_CAMPAIGN_ROOT` — recoverable cold). The skill
dir is not derivable: the resolved absolute path exists only in the (compacted) context
(SKILL.md:11-22), plugin install roots vary, and `paths.py` can't bootstrap it because
paths.py lives *inside* the skill dir. So D2-7's "re-read the behavior contract" fix can
strand itself: post-compaction, the model knows it should re-read SKILL.md and no longer
knows where it is.

**Rider for F:** load step 5 (SKILL-commands.md:98) already writes
`{"name": "<campaign>"}` to `<runtime-dir>/active-campaign.json`, and the runtime dir IS
derivable (`<data-root>/.runtime`, paths.py:81-86). Extend the payload:
`{"name": "...", "skill_dir": "<absolute path>"}` — `autosave_checkpoint.py` reads only
the `name` key (autosave_checkpoint.py:80) and tolerates extras. D2-7's ladder step then
ends: *"if the skill-dir path itself was lost, read it from
`<data-root>/.runtime/active-campaign.json`."* One caveat to state in the D2-7 prose: a
disk re-read returns `${CLAUDE_SKILL_DIR}` **unexpanded** (the substituted copy exists
only at skill load — SKILL.md:15-21 applies to SKILL.md itself), fine for behavior rules,
not a source for runnable paths.

### E7 — Continuity Archive compression runs *before* the graph sweep it depends on

**Claim.** In the save procedure, log archival + compression (SKILL-commands.md:391-420)
comes **before** the relationship-shift sweep (:422-438). The compression rule drops any
bullet that restates relational state the graph "holds canonically" (:402-407) — but at
the moment of compression, *this session's* relational facts are not yet in the graph:
the sweep hasn't run, it's approval-gated (`y / pick / skip`, :430-434), and `skip` means
they never arrive. A bullet dropped on the assumption is the compass split-brain
([A] register) made concrete: fact deleted from the archive *and* absent from the graph.
The adjacent contradiction: micro-save appends graph edges *silently* (SKILL.md:223,
"silently flush … append any new relationships") while the sweep rule says "Never write
proposed edges silently" (SKILL-commands.md:436).

**Proposed fix (prose):**
1. Reorder: run the graph sweep before log archival (swap the two blocks; archival's
   "after session count > 3" gate is unaffected).
2. Harden the drop-rule (:402): *"drop a relational bullet only when its edge is
   **confirmed present** — it appeared in the just-approved sweep batch or an earlier
   session's graph. If in doubt, keep the bullet."*
3. Resolve the silent-write contradiction: micro-save appends only edges **explicitly
   narrated on-screen this scene** (the live `add-edge` discipline,
   SKILL-commands.md:788); anything inferential waits for the approval sweep.

### E8 — An authored campaign that finishes its spine has nowhere to go

**Claim.** When the final beat completes, `beat complete` step 2 sets
`current_beat: null` and appends an Arc History line (SKILL-commands.md:352-356) — and
stops. The arc-continuation flow (ask, then `arc new` or convert to sandbox) exists only
inside dynamic `arc advance` (SKILL-commands.md:703-706), and `arc new` is gated
`type: dynamic` (:698). So the flagship path ends its campaign in limbo: no continuation
offer, no sandbox conversion, an arc block no steering gate matches (even post-D2-1-fix,
the dynamic rules assume a live beat window). Same genus as D2-1 — the authored path
lacks wiring the legacy path has.

**Proposed fix (prose, `beat complete` final-beat branch):** mirror :703-706 — *"Ask:
'The arc is complete. Continue with a new arc? [y/n]' — yes → run `/dm:dnd arc new`
(note: the successor arc is generated in **dynamic** format from the resolved world; the
spine was a one-arc artifact); no → set `type: sandbox`."*

### E9 — C8 handoff extension: `arc revise` relocation also strands world.md Adventure Nodes and the graph

**Claim.** The revise flow (SKILL-commands.md:723-733) touches beat fields,
`revision_log`, `steering_notes` — nothing else. If the revision relocates a beat or
re-assigns which faction/NPC drives it: the `world.md → ## Adventure Nodes` web still
points at the old geography (nodes "move, don't disappear" is Standard 10 table
discipline, SKILL.md:132, with no revise-time hook), and graph place/NPC edges still
encode the old relationships — so `scene-context`, which load step 6 and the ladder treat
as authoritative for who-relates-to-whom (SKILL-commands.md:122), **actively reasserts
the pre-revision world after a compaction**. Scope answer for the C8 handoff:
`world-nodes.md` itself is safe — it's structured-only and revise is a no-op for
structured campaigns (SKILL-commands.md:698); the exposure is world.md + graph.json on
dynamic/authored.

**Proposed fix (prose):** extend C8's planned revise-step line to: *"If the revision
relocates a scene or changes which NPC/faction drives it, also: update the affected
`world.md → ## Adventure Nodes` entries, propose the matching graph
`add-edge`/`close-edge` calls in the same confirmation, and refresh the shopping lists +
re-run `render_assets.py` (C8)."* One consolidated step instead of three drifting ones.

### E10 — Micro-save moves faction stances without their deeds

**Claim.** The Deed rule — "never change a faction's stance without recording the cause"
(SKILL-commands.md:382-384) — lives only in the full-save procedure. The micro-save flush
list (SKILL.md:223) updates Live State Flags (which contain the stances) but does not
mention the Deeds ledger. So the designed sequence "stance shifts mid-session at a scene
boundary → flags flushed → compaction → save" produces a stance whose cause the save-time
deed must be reconstructed from compacted memory — the audit trail's one job
(templates/state.md:28-30, ADR-0016 decision B) failing exactly when it's needed.

**Proposed fix (prose, one clause in SKILL.md:223's flush list):** *"…update `## Live
State Flags` (and if a faction stance changed, append its Deeds line in the same flush —
the ledger is append-only and one line)…"*

## LOW severity

### E11 — Recap loop wiring (open-question answer): diff-at-load wins, and one call does everything

The timing contradiction ([A] item 3) resolves cleanly once you notice `diff`
**auto-advances the baseline** (SKILL-scripts.md:215-218). Sequence: diff at load S2
compares the S1 baseline against current state (= end of S1), prints last session's
mechanical changes, and rolls the baseline to start-of-S2 — self-chaining forever.
Snapshot-at-end (SKILL-commands.md:823) is functionally wrong: an end-of-session baseline
diffed at next load reports nothing (nothing changes between sessions). **Fix:** insert
load step 6.5: *"run `session_recap.py diff --campaign <name>` and inject the output line
as the mechanical half of the step-7 recap (first-ever load: run `snapshot` instead —
no baseline exists yet)"*; rewrite SKILL-commands.md:822-826 to match and delete "Take a
snapshot at `/dm:dnd end`". This also dissolves the duplicate-`recap` confusion
([A] item 4): the deterministic subcommands become load machinery; `/dm:dnd recap`
(:685) stays the narrative command.

### E12 — `autosave_checkpoint.py` docstring promises a tail health-check the code never performs

autosave_checkpoint.py:10 claims the snapshot step "verifies the campaign's
`session_tail.json` is healthy." The implementation (`snapshot()`,
autosave_checkpoint.py:166-176) copies state.md and nothing else — no reference to the
tail anywhere in the code. Either implement (a `json.loads` + non-empty-list check with a
warning line, ~6 lines) or cut the sentence. Related: the every-turn
`<runtime>/<camp>.autocheckpoint.md` snapshot has **zero readers and no documented
recovery procedure** (only a test asserts it exists, test_autosave_checkpoint.py:123).
One line in SKILL-scripts.md:307-324 closes it: *"Recovery: if state.md is lost or
mangled, the latest turn-level snapshot is at
`<runtime-dir>/<campaign>.autocheckpoint.md` — copy it back over state.md."*

### E13 — tracker.html keeps refreshing a dead combat

Combat end (SKILL-commands.md:663) clears `## Active Combat` and tracker.py state but
never re-renders tracker.html — the meta-refresh dashboard serves the final round to the
host's browser indefinitely, displaying an active fight that ended an hour ago. **Fix:**
combat-end step gains: *"re-run `render_tracker.py` once with the final state and a
'combat ended' marker (or clear tracker.html) so the host's dashboard doesn't show a live
fight that's over."*

### E14 — `supersede-edge` (open-question answer): intended surface, just undocumented

campaign_graph.py:236-245 implements it deliberately — hard-retcon semantics, explicitly
distinguished from `close-edge` ("close-edge ends a state cleanly … supersede-edge says
the original edge was wrong"), with audit-trail preservation and active-query filtering
(campaign_graph.py:155-159). This is exactly the tool for correcting a mis-extracted or
mis-narrated edge, and the graph section documents everything else. **Fix:** add a
`### /dm:dnd graph supersede-edge --id <edge> [--by <edge>] [--reason ...]` block after
close-edge (SKILL-commands.md:758-759 region) carrying that close-vs-supersede
distinction.

### E15 — A pending level-up marker can strand across sessions

`beat complete` step 3 stamps `⚠ LEVEL UP PENDING (Level N)` then runs level-up passes
(SKILL-commands.md:357-361). If the session ends between the stamp and the passes (host
quits at the beat boundary — a natural stopping point), the marker persists on the sheet.
Load reads all sheets (SKILL-commands.md:98) so the text enters context, but no step
instructs acting on it, and the spine's banding math assumes the level was applied. 
**Fix:** load step 5 one-liner: *"If any character sheet carries `⚠ LEVEL UP PENDING`,
surface it and run `/dm:dnd level up` before play begins."*

## What already works (so F doesn't regress it)

- **schema.py's status-coherence gate is real and subtle** — single-`current`
  enforcement plus the monotonic resolved→current→pending phase check
  (schema.py:99-117) catches impossible play-states, not just bad authoring. E4's
  write-back leans on it deliberately.
- **The autosave hook is carefully engineered:** the `stop_hook_active` no-loop guard,
  section-scoped flag parsing (won't match "autosave" in prose), pure testable
  `decide()`, and best-effort I/O that can never kill a turn
  (autosave_checkpoint.py:149-176).
- **The lazy-load layering is consistent end to end** — load step 5's do-NOT-read list
  (SKILL-commands.md:100-110) matches a specific on-demand trigger for every excluded
  file; nothing is orphaned on the read side *except* the session tail (E2).
- **Code/data root separation** (SKILL.md:145-175, paths.py) is clean: campaign data
  survives plugin updates, and the E6 fix rides on the runtime dir this separation
  already provides.
- **close-edge vs supersede-edge semantics** (E14) show genuine data-modeling care —
  ended-truth vs never-was-truth is a distinction most campaign tools never make.
- **Tail-verification at `end`** (SKILL-commands.md:457) means the write side of the
  session tail is solid; E2 only has to add readers.
